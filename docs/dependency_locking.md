# Dependency Locking (JTN-616)

InkyPi uses [uv](https://docs.astral.sh/uv/) to produce a universal, hash-pinned
lockfile that covers every platform we support (Linux x86_64/aarch64/armv7l/armv6l
and macOS arm64/x86_64) from a single resolution.

## Source of Truth

1. **`pyproject.toml`** — declares top-level runtime dependencies in
   `[project.dependencies]`.
2. **`uv.lock`** — universal cross-platform lockfile generated by `uv lock`.
3. **`install/requirements.txt`** — hash-pinned requirements file exported from
   `uv.lock` via `uv export`. This is the file consumed by `install/install.sh`
   with `--require-hashes` (preserves the JTN-516 supply-chain guarantee).

During the migration window (JTN-616), `install/requirements.in` is retained as
a legacy reference. It is **not** the source of truth any longer — edit
`pyproject.toml` instead.

## How to Regenerate

After changing a dependency in `pyproject.toml`:

```bash
# 1. Update uv.lock from pyproject.toml
uv lock

# 2. Re-export the hash-pinned requirements file used by install.sh
uv export --format requirements.txt --no-dev --no-emit-project \
    --output-file install/requirements.txt
```

Commit both `uv.lock` and `install/requirements.txt`.

To verify everything is in sync locally (the same check that runs in CI):

```bash
bash scripts/check_requirements_drift.sh
```

## Why `uv lock` (not `pip-compile`)

The previous `pip-compile` based workflow caused chronic lockfile-drift failures:

| Issue | pip-compile | uv lock |
| --- | --- | --- |
| Python version coupling | Resolution is Python-version specific (3.12 vs 3.13 produced different trees) | Universal resolution — Python-version agnostic |
| `sys_platform` markers | Cannot resolve `sys_platform == "linux"` packages when run on macOS; required a manually-maintained Linux-only block appended to `requirements.txt` | Resolves all markers together; Linux-only packages appear naturally with the marker preserved |
| Multi-arch wheel hashes | Only produced hashes for the current host architecture; Pi arm64/armv7l/armv6l hashes were hand-managed | Emits hashes for every wheel that matches the universal resolution, including all supported architectures |
| Speed | Slow on repeated runs | ~10x faster, single binary |

The universal lockfile means a Mac developer, a Linux CI runner, and a Pi Zero
2 W all see the exact same resolved versions with hashes covering their wheels.

## Install Path Unchanged

`install/install.sh` still installs from `install/requirements.txt` with
`--require-hashes`. No behaviour change on the Pi — only the tool that
regenerates the file is different.

## Dev Requirements

`install/requirements-dev.txt` is **not** part of this migration. It still
tracks `install/requirements-dev.in` via pip-compile. Migrating dev deps is
deferred to a follow-up (Phase 3 of JTN-616).
